# AI-Ready Layers

How this site makes itself legible to AI agents and crawlers — markdown twins, `llms.txt`/
`llms-full.txt`, the machine-readable CV, and edge detection — per the V3b addendum, commits 2–3,
with PT-BR twin coverage added by the V3d refinement addendum.

## 1. Markdown twins

Every content page has a clean, front-matter-free markdown sibling, generated at build time from
the exact same `ContentRepository` data (and, for MDX pages, the same raw entry body) the HTML
page renders — so redaction rules apply identically; there is no second copy of the content to
drift out of sync.

| HTML page | Markdown twin |
|---|---|
| `/` | `/home.md` |
| `/experience` | `/experience/index.md` |
| `/experience/dossier` | `/experience/dossier.md` |
| `/speaking` | `/speaking/index.md` |
| `/contact` | `/contact/index.md` |
| `/projects/<slug>` | `/projects/<slug>.md` |
| `/writing/<slug>` | `/writing/<slug>.md` |

Generation logic lives in `src/core/services/markdown-twin.ts` (pure functions, one per page
shape); the thin `.md.ts` Astro endpoints (e.g. `src/pages/experience/index.md.ts`) call these and
set `Content-Type: text/markdown; charset=utf-8`. All are `prerender = true` — zero server cost,
built once at `pnpm build` time like every other static route.

Every page with a twin advertises it via `<link rel="alternate" type="text/markdown" href="...">`
in `<head>` (`src/layouts/BaseLayout.astro`'s `markdownTwin` prop) — the standard, passive
discovery mechanism a crawler or agent can follow without any special-case logic.

### 1.1 PT-BR twins (V3d refinement addendum)

Every `/pt/` page that renders in Portuguese (per the V3d addendum's i18n scope) now has its own
markdown twin, generated by the **same** `markdown-twin.ts` functions the EN twins use — each
function takes an optional `locale: Locale = 'en'` parameter that swaps section-heading labels
(`## Credibility` → `## Credibilidade`, etc.) and, for content methods, gets forwarded into the
matching `ContentRepository` locale-aware call (`getHero('pt-br')`, `getRoles('pt-br')`, ...).
There is no separate PT generator — this is the single-definition policy applied to the twin
layer, closing the TODO that previously lived in `src/pages/llms.txt.ts` and this file.

| HTML page | Markdown twin |
|---|---|
| `/pt/` | `/pt/home.md` |
| `/pt/experience` | `/pt/experience/index.md` |
| `/pt/experience/dossier` | `/pt/experience/dossier.md` |
| `/pt/speaking` | `/pt/speaking/index.md` |
| `/pt/contact` | `/pt/contact/index.md` |
| `/pt/projects` | `/pt/projects/index.md` |
| `/pt/writing` | `/pt/writing/index.md` |

Two of these — `/pt/projects/index.md` and `/pt/writing/index.md` — are also the *first* twins for
their respective listing shape: the EN `/projects` and `/writing` listing pages never had a twin
of their own (only individual `/projects/<slug>` and `/writing/<slug>` pages did). They're
generated by two new functions, `projectsListingToMarkdown` and `writingListingToMarkdown`, that
mirror what the PT HTML listing pages themselves already do (V3d addendum decision): PT chrome
around an EN-titled project/article list, since project deep-dives and articles stay English-only
in v1. The twin makes that explicit with a one-line note rather than silently mixing languages.

Individual project deep-dives (`/projects/<slug>`) and articles (`/writing/<slug>`) do not get PT
twins — they don't render in Portuguese at all yet (V3d addendum scope decision), so there's
nothing to twin.

Every `/pt/` page with a twin advertises it the same way the EN pages do:
`<link rel="alternate" type="text/markdown" href="/pt/...">` in `<head>`.

## 2. `llms.txt` and `llms-full.txt`

- **`/llms.txt`** (`src/pages/llms.txt.ts`) — the emerging convention: a short, markdown-link
  index of key pages, all project/article entries, and now every markdown twin (EN and PT-BR)
  plus the machine-readable-CV and `llms-full.txt` links, so an agent parsing `llms.txt` alone can
  discover every AI-readable surface on the site, in both languages.
- **`/llms-full.txt`** (`src/pages/llms-full.txt.ts`) — every markdown twin concatenated into one
  document: the EN set (Home, Journey, Dossier, Speaking, Contact, every project, every published
  article), then a `pt-BR` section with the seven PT twins above, separated by `---` rules, for
  agents that prefer fetching a single URL over crawling the twin list. Built from the same pure
  `markdown-twin.ts` functions for both languages — no separate content path.

## 3. `/api/cv.json` — JSON Resume

A static, prerendered JSON document at `/api/cv.json` following the
[JSON Resume](https://jsonresume.org) schema (`$schema`, `basics`, `work`, `education`, `skills`,
`languages`, `certificates`), built by `src/core/services/json-resume.ts` from roles + the dossier
+ site singletons — the same redacted `ContentRepository` data every HTML page reads. Linked from
`llms.txt` and from the dossier page footer ("Machine-readable CV: `/api/cv.json`"). `prerender =
true`, so it costs nothing at request time.

This same JSON is also what the MCP server's `get_cv` tool returns — see `docs/mcp.md`. Where the
layers above are passive (a crawler fetches a URL), MCP is active: an agent calls a typed tool with
validated input and gets structured data back, plus a grounded chatbot (`ask_felipe`) and a way to
leave Felipe a message (`leave_message`).

## 4. Redaction-gate coverage

`scripts/redaction-gate.ts`'s `TEXT_EXTENSIONS` set already included `.md`, `.json`, and `.txt`
before this work — its `dist/` scan pass (run after `pnpm build`) therefore already covers every
new artifact type (markdown twins, `llms-full.txt`, `cv.json`) with no changes needed to the
walker itself. Verified by running `pnpm redaction-gate` after a full build with all of V3b's new
routes present (clean pass), and by two new unit tests
(`src/core/services/__tests__/redaction-gate.test.ts`) proving `scanText` — the underlying scanner
— catches a forbidden figure inside markdown-twin-shaped text and inside a JSON-Resume-shaped
`highlights` array.

## 5. Edge AI-agent detection — what works and what doesn't

`src/middleware.ts` implements the addendum's ask: match a known bot User-Agent (`GPTBot`,
`ClaudeBot`, `Claude-User`, `anthropic-ai`, `PerplexityBot`, `Google-Extended`, `CCBot`,
`meta-externalagent`, `Bytespider` — maintained in `src/config/ai-agents.ts`) or an `Accept` header
preferring `text/markdown`, and 302-redirect to the page's markdown twin.

**It does not currently redirect any real content page, in dev or in production — this is a
discovered, documented limitation, not a bug:**

1. **Astro strips request headers for every prerendered page, full stop — not a Cloudflare
   quirk.** This site's entire content surface (`/`, `/experience`, `/projects/*`, `/writing/*`,
   `/speaking`, `/contact`, `/experience/dossier`) is prerendered by default under `output:
   'static'`. Astro's own request constructor
   (`node_modules/astro/dist/core/request.js`) sets `headers: undefined` whenever
   `isPrerendered` is true, and the dev server (`vite-plugin-astro-server/route.js`) passes
   `isPrerendered: route.prerender` into that constructor *before* middleware runs — confirmed by
   tracing both files and by a live request against `pnpm dev`, which returned `null` for
   `request.headers.get('user-agent')` on `/experience` even with a matching bot UA sent via curl.
   Astro's own console warning says the same thing: *"`Astro.request.headers` is not available on
   prerendered pages ... make sure the page is server-rendered."* This reproduces identically
   whether or not Cloudflare is involved.
2. **On top of that, Cloudflare Pages routing would block it anyway.** `@astrojs/cloudflare`
   12.6.9, combined with this project's Pages-style `wrangler.jsonc` (`pages_build_output_dir:
   "dist"`), generates `dist/_routes.json` at build time with every prerendered route explicitly
   **excluded** from Worker invocation (verified directly: `/`, `/contact/*`, `/experience/*`,
   `/projects/*`, `/writing/*`, `/speaking/*`, `/home.md`, `/llms.txt` etc. all appear under
   `"exclude"` in the generated file) — Cloudflare Pages serves those straight from its static
   asset store, and this Worker (including its middleware) never sees the request at all.
   `run_worker_first` — the wrangler config field that would force such routes through the Worker
   first — belongs to the newer **Workers with Static Assets** `assets` config block
   (`directory`/`binding`/`html_handling`/`not_found_handling`/`run_worker_first`), a different
   deployment model than `pages_build_output_dir`. The two are mutually exclusive shapes in
   wrangler's schema; `@astrojs/cloudflare` 12.6.9's own build output targets the classic Pages
   model. Adopting the newer model would be a deployment-model migration for the whole project,
   not a config tweak — out of scope here, and not attempted.

**The only place the middleware's logic can technically fire** is a route that's already
`export const prerender = false` — `/api/contact` and `/api/chat` — and neither is an HTML content
page a bot would navigate to, so `markdownTwinFor()` has nothing to match there either. The
middleware is still wired in (`src/middleware.ts`) because it's correct, harmless, zero-cost code
that would start working the moment any specific route is deliberately flipped to
`prerender = false` — a real option for the future, but one that trades CDN-static serving (this
site's actual $0/free-tier model) for that route, and is not a decision this change makes
unilaterally.

**The guarantee for AI agents today is entirely the passive layer**: the `<link rel="alternate"
type="text/markdown">` tag and the `llms.txt`/`llms-full.txt` twin listing (section 1–2 above) —
both reach an agent with zero dependency on anything running at request time, in dev or in
production, on any host.

`tests/e2e/ai-agent-redirect.spec.ts` asserts the actual current behavior (bot UA and
markdown-preferring `Accept` both still get the normal HTML page; the twin is separately fetchable
directly) rather than asserting a redirect that cannot happen — so a future change that "fixes"
the middleware without first addressing point 1 above would be caught by a real regression, not
masked by an assertion on behavior the framework doesn't allow.
